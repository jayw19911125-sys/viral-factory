# 爆款短影音拆解工廠 v3.0：反方極限審查與突破性優化報告

> **審查基準：** 以「接班人式保姆級獲利教程」與「降本增效利益最大化」為最高標準，拒絕任何「堪用就好」的妥協。系統必須在無人干預的情況下，100% 精準地將市場流量密碼轉化為好創的資產。

經過對 `/home/ubuntu/viral_factory` 目錄下所有程式碼、執行日誌與產出的深度掃描，我發現目前的系統雖然能跑，但本質上充滿了**資料斷層、虛假自動化與架構脆弱性**。這不是一個成熟的「工廠」，而是一個隨時會停擺的「實驗室」。

以下是徹底審查後揭露的四大核心缺陷，以及超越預期的極致優化方案。

---

## 🛑 第一層缺陷：虛假的自動化與資料斷層（嚴重阻斷）

目前系統在資料傳遞上存在致命的斷層，導致所謂的「自動分配」與「週報統計」完全是幻覺。

### 1. 欄位不對齊導致的「未命名」災難
在 `viral_factory.py` 中，GPT 輸出的 JSON 鍵值為中文的 `"影片標題或主題"`，但在 `script_distributor.py` 中，系統卻試圖讀取英文的 `"title"`。這導致所有被分配到 Notion 的腳本全部變成「未命名腳本」，且遺失了大量關鍵資訊（如 `hook_type`、`neuroscience` 等），讓庫房淪為垃圾堆。

### 2. Notion MCP 工具名稱錯誤導致週報癱瘓
`weekly_manual_updater.py` 與 `weekly_report.py` 試圖呼叫 `notion-query-database`，但實際可用的 MCP 工具清單中，根本沒有這個工具（正確應為 `notion-fetch` 或 `notion-search`）。這導致每週一的統計數據永遠是 0，GPT 產出的週報摘要完全基於錯誤的空資料進行「幻覺分析」。

### 3. Meta 廣告的「永遠模擬」
`meta_ads_fetcher.py` 中充斥著 `_mock` 與 `模擬` 的硬編碼。所謂的「Meta 廣告按產業輪流」，實際上只是在跑迴圈印出假資料，根本沒有接入真實的 Meta API 或爬蟲，這讓雙來源系統直接廢了一半。

---

## 🛑 第二層缺陷：架構脆弱性與單點故障風險

系統缺乏錯誤恢復機制，任何一個小環節出錯都會導致整個流程崩潰。

### 1. Whisper API 的單點依賴
依賴外部 OpenAI API 進行 Whisper 語音轉文字，一旦 Token 失效或額度耗盡，整個拆解流程直接停擺（如 6/19 日誌所示，6 支影片全部失敗）。雖然已暫時替換為 `manus-speech-to-text`，但缺乏自動降級或重試機制。

### 2. GitHub Token 的維護地雷
排程每次執行前都試圖 `git pull`，但 GitHub Token 是寫死且會過期的。一旦過期，排程就會報錯。這不是真正的自動化部署，而是埋了一顆定時炸彈。

### 3. S3 上傳的環境變數缺失
手冊週更最後一步試圖將 PDF 上傳至 S3，但 `.user_env` 中根本沒有設定 AWS 憑證，導致上傳必定失敗，Slack 通知中的下載連結永遠無效。

---

## 🛑 第三層缺陷：內容產出的平庸與「靠感覺」

目前的評分機制與拆解邏輯，仍然停留在表層，無法真正指導剪輯與企劃。

### 1. 評分標準的模糊性
`script_scorer.py` 的評分 prompt 過於主觀（例如：「鉤子明確且視覺錘突出」），缺乏量化的數據支撐（如完播率預估、互動率指標）。這導致 3 分與 4 分的界線模糊，無法精準篩選出真正的爆款。

### 2. 缺乏「反向工程」的指導意義
拆解出的「腳本結構」只是文字描述，阿韋（剪輯師）拿到後依然需要自己思考如何套用到現有素材。系統沒有將拆解結果轉化為「可直接執行的剪輯時間軸（Timeline）」。

---

## 🚀 突破性極致優化方案（v4.0 藍圖）

為了讓這套系統真正成為好創的「印鈔機」，我提出以下超越現狀的極致優化方案。這不僅是修復 Bug，而是將系統推向完全自主運作的境界。

### 階段一：資料鏈條的絕對精準（立即執行）
1. **重構 JSON Schema**：強制使用 Pydantic 或 TypedDict 定義 GPT 的輸出結構，確保 `viral_factory.py` 與 `script_distributor.py` 之間的欄位 100% 對齊，徹底消滅「未命名」。
2. **替換 Notion MCP 呼叫**：將所有錯誤的 `notion-query-database` 替換為官方正式支援的 Notion API (透過 `notion-client` 套件) 或正確的 MCP 工具，確保週報數據絕對真實。
3. **完善環境變數檢查**：在腳本啟動時，強制檢查所有必要的 API Key（OpenAI, Notion, Slack, AWS），若有缺失立即發送高優先級 Slack 警告給子權，而不是默默失敗。

### 階段二：雙來源系統的真實落地
1. **Meta 廣告真實爬取**：捨棄模擬資料，整合 Apify 或正式的 Meta Ad Library API，抓取真實投放超過 30 天且互動極高的廣告素材。
2. **引入 IG Reels 監控**：補齊目前缺失的 IG Reels 數據，利用 session cookie 或爬蟲技術，完成 TikTok、Meta、IG 的三足鼎立。

### 階段三：剪輯師的「保姆級」賦能（能力躍升）
1. **自動生成剪輯 Timeline**：拆解腳本時，不再只給文字，而是由 GPT 自動產出對應的 **剪輯指令表**（例如：`00:00-00:03 畫面：特寫 / 音效：叮 / 字幕：大膽宣言`），讓阿韋可以直接照表操課，將效率再提升 5 倍。
2. **素材庫的自動關聯**：當系統判斷某支影片是「IP型」時，自動從好創現有的素材庫中，推薦可用的 B-roll 或畫面，實現半自動化剪輯前置作業。

### 階段四：商業價值的閉環
1. **自動化報價單生成整合**：當高分腳本（4-5分）入庫時，自動計算若要為客戶製作類似影片的成本與報價，並同步寫入 Monday.com，讓營運長（子權）能直接拿著報價單去談業務。
2. **全局知識地圖同步**：將所有拆解出來的「視覺錘」與「語言釘」定期更新至好創的全局知識地圖，成為全公司共用的行銷資產。

---

**結論：**
目前的 v3.0 系統是一個充滿潛力的半成品，但被脆弱的資料傳遞與虛假自動化拖累。只要按照上述方案進行徹底重構，這套系統將真正實現「降本增效」，讓好創的短影音產能與質量達到市場頂尖水準。
